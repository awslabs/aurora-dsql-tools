# dsql-lint Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a Rust CLI tool that parses SQL files and reports DSQL compatibility errors and warnings with suggested fixes.

**Architecture:** Uses `sqlparser-rs` to parse SQL into a full AST (PostgreSQL dialect), then walks the AST applying manually-curated DSQL compatibility rules. Each rule produces an ERROR or WARNING with a human-readable suggestion. The tool is both a standalone CLI (`dsql-lint migrate.sql`) and a library crate that the aurora-dsql-mcp-server can import.

**Tech Stack:** Rust, sqlparser-rs (v0.61+), clap (CLI args)

## Output Format

```
migration.sql:5: ERROR — FOREIGN KEY constraints not supported in DSQL
  → Remove FOREIGN KEY and implement referential integrity in application layer
migration.sql:12: ERROR — CREATE INDEX must use ASYNC
  → CREATE INDEX ASYNC idx_name ON table(col)
migration.sql:20: ERROR — SERIAL type not supported in DSQL
  → Use INTEGER GENERATED BY DEFAULT AS IDENTITY (CACHE 1)
migration.sql:30: WARNING — Table 'orders' missing tenant_id column
  → Add tenant_id VARCHAR(255) NOT NULL for multi-tenant isolation
```

Exit code: 0 if no errors (warnings OK), 1 if any errors found.

## Rules Reference

### ERRORS (will fail on DSQL)

| ID | Pattern | Suggestion |
|----|---------|------------|
| E001 | `FOREIGN KEY` / `REFERENCES` constraint | Remove and implement referential integrity in application layer |
| E002 | `CREATE INDEX` without `ASYNC` | Add `ASYNC` keyword: `CREATE INDEX ASYNC ...` |
| E003 | `SERIAL` column type | `INTEGER GENERATED BY DEFAULT AS IDENTITY (CACHE 1)` |
| E004 | `BIGSERIAL` column type | `BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)` |
| E005 | `SMALLSERIAL` column type | `SMALLINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)` |
| E006 | `JSON` / `JSONB` column data type | Use `TEXT` and serialize with JSON.stringify |
| E007 | `TRUNCATE` statement | Use `DELETE FROM table_name` instead |
| E008 | `CREATE TEMP TABLE` / `TEMPORARY` | Use regular tables or application-level caching |
| E009 | Array column types (`INT[]`, `TEXT[]`, etc.) | Use `TEXT` and store as comma-separated values |
| E010 | `CREATE TRIGGER` | Implement trigger logic in application layer |
| E011 | `CREATE EXTENSION` | Extensions not supported in DSQL |
| E012 | `PARTITION BY` in CREATE TABLE | Omit — DSQL manages distribution automatically |
| E013 | `CREATE FUNCTION` / `CREATE PROCEDURE` (PL/pgSQL) | Implement in application layer |

### WARNINGS (best practice violations)

| ID | Pattern | Suggestion |
|----|---------|------------|
| W001 | `CREATE TABLE` missing `tenant_id` column | Add `tenant_id VARCHAR(255) NOT NULL` for multi-tenant isolation |
| W002 | `ALTER TABLE ADD COLUMN` with inline `DEFAULT` or `NOT NULL` | Split into separate steps: ADD COLUMN, then UPDATE, then ALTER COLUMN |

use clap::Parser;
use std::process;

#[derive(Parser)]
#[command(name = "dsql-lint")]
#[command(about = "Lint SQL files for Aurora DSQL compatibility")]
struct Args {
    /// SQL files to lint
    files: Vec<String>,
}

fn main() {
    let args = Args::parse();

    if args.files.is_empty() {
        eprintln!("Usage: dsql-lint <file.sql> [file2.sql ...]");
        process::exit(1);
    }

    let mut total_errors = 0;

    for path in &args.files {
        let sql = std::fs::read_to_string(path).unwrap_or_else(|e| {
            eprintln!("Error reading '{path}': {e}");
            process::exit(1);
        });

        let diagnostics = dsql_lint::lint::lint_sql(&sql);
        for d in &diagnostics {
            let severity = if d.is_error { "ERROR" } else { "WARNING" };
            eprintln!("{path}:{}: {severity} — {}", d.line, d.message);
            eprintln!("  → {}", d.suggestion);
        }
        total_errors += diagnostics.iter().filter(|d| d.is_error).count();
    }

    if total_errors > 0 {
        eprintln!("\n{total_errors} error(s) found.");
        process::exit(1);
    } else if !args.files.is_empty() {
        eprintln!("All statements compatible with DSQL.");
    }
}
```

//! Core linting engine: parse SQL → walk AST → apply rules → collect diagnostics.

use sqlparser::ast::Statement;
use sqlparser::dialect::PostgreSqlDialect;
use sqlparser::parser::Parser;

use crate::rules;

/// A single diagnostic produced by a lint rule.
#[derive(Debug, Clone)]
pub struct Diagnostic {
    /// 1-based line number in the input (best effort from sqlparser location).
    pub line: usize,
    /// Rule ID, e.g. "E001".
    pub rule_id: String,
    /// Human-readable message.
    pub message: String,
    /// Suggested fix.
    pub suggestion: String,
    /// true = ERROR (will fail on DSQL), false = WARNING (best practice).
    pub is_error: bool,
}

/// Lint a SQL string and return all diagnostics.
pub fn lint_sql(sql: &str) -> Vec<Diagnostic> {
    let dialect = PostgreSqlDialect {};
    let statements = match Parser::parse_sql(&dialect, sql) {
        Ok(stmts) => stmts,
        Err(e) => {
            return vec![Diagnostic {
                line: 1,
                rule_id: "E000".to_string(),
                message: format!("Failed to parse SQL: {e}"),
                suggestion: "Fix the SQL syntax and try again.".to_string(),
                is_error: true,
            }];
        }
    };

    let mut diagnostics = Vec::new();
    for stmt in &statements {
        rules::check_statement(stmt, sql, &mut diagnostics);
    }
    diagnostics
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_valid_create_table_no_errors() {
        let sql = "CREATE TABLE orders (id UUID PRIMARY KEY, tenant_id VARCHAR(255) NOT NULL, amount DECIMAL(10,2));";
        let diags = lint_sql(sql);
        let errors: Vec<_> = diags.iter().filter(|d| d.is_error).collect();
        assert!(errors.is_empty(), "Expected no errors, got: {:?}", errors);
    }

    #[test]
    fn test_parse_error_returns_diagnostic() {
        let sql = "NOT VALID SQL AT ALL ???";
        let diags = lint_sql(sql);
        assert!(!diags.is_empty());
        assert_eq!(diags[0].rule_id, "E000");
    }
}

/// Check a statement for DSQL error-level incompatibilities.
pub fn check(stmt: &Statement, raw_sql: &str, diagnostics: &mut Vec<Diagnostic>) {
    check_create_table(stmt, diagnostics);
    check_create_index(stmt, raw_sql, diagnostics);
}

fn check_create_table(stmt: &Statement, diagnostics: &mut Vec<Diagnostic>) {
    let Statement::CreateTable(create) = stmt else { return };

    for column in &create.columns {
        // E003/E004/E005: SERIAL types
        let type_str = column.data_type.to_string().to_uppercase();
        if type_str == "SERIAL" || type_str == "SERIAL4" {
            diagnostics.push(Diagnostic {
                line: 1,
                rule_id: "E003".to_string(),
                message: format!("SERIAL type not supported in DSQL (column '{}')", column.name),
                suggestion: format!(
                    "Use: {} INTEGER GENERATED BY DEFAULT AS IDENTITY (CACHE 1)",
                    column.name
                ),
                is_error: true,
            });
        } else if type_str == "BIGSERIAL" || type_str == "SERIAL8" {
            diagnostics.push(Diagnostic {
                line: 1,
                rule_id: "E004".to_string(),
                message: format!("BIGSERIAL type not supported in DSQL (column '{}')", column.name),
                suggestion: format!(
                    "Use: {} BIGINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)",
                    column.name
                ),
                is_error: true,
            });
        } else if type_str == "SMALLSERIAL" || type_str == "SERIAL2" {
            diagnostics.push(Diagnostic {
                line: 1,
                rule_id: "E005".to_string(),
                message: format!("SMALLSERIAL type not supported in DSQL (column '{}')", column.name),
                suggestion: format!(
                    "Use: {} SMALLINT GENERATED BY DEFAULT AS IDENTITY (CACHE 1)",
                    column.name
                ),
                is_error: true,
            });
        }

        // E001: FOREIGN KEY via column REFERENCES
        for option in &column.options {
            if let ColumnOption::ForeignKey { .. } = &option.option {
                diagnostics.push(Diagnostic {
                    line: 1,
                    rule_id: "E001".to_string(),
                    message: format!(
                        "FOREIGN KEY (REFERENCES) not supported in DSQL (column '{}')",
                        column.name
                    ),
                    suggestion: "Remove REFERENCES and implement referential integrity in application layer.".to_string(),
                    is_error: true,
                });
            }
        }
    }

    // E001: FOREIGN KEY via table-level constraint
    for constraint in &create.constraints {
        if matches!(constraint, TableConstraint::ForeignKey { .. }) {
            diagnostics.push(Diagnostic {
                line: 1,
                rule_id: "E001".to_string(),
                message: "FOREIGN KEY table constraint not supported in DSQL".to_string(),
                suggestion: "Remove FOREIGN KEY and implement referential integrity in application layer.".to_string(),
                is_error: true,
            });
        }
    }
}

/// W001: CREATE TABLE missing tenant_id column.
fn check_missing_tenant_id(stmt: &Statement, diagnostics: &mut Vec<Diagnostic>) {
    let Statement::CreateTable(create) = stmt else { return };

    let has_tenant_id = create.columns.iter().any(|col| {
        col.name.value.to_lowercase() == "tenant_id"
    });

    if !has_tenant_id {
        let table_name = &create.name;
        diagnostics.push(Diagnostic {
            line: 1,
            rule_id: "W001".to_string(),
            message: format!("Table '{table_name}' missing tenant_id column"),
            suggestion: "Add tenant_id VARCHAR(255) NOT NULL for multi-tenant isolation.".to_string(),
            is_error: false,
        });
    }
}

/// W002: ALTER TABLE ADD COLUMN with inline DEFAULT or NOT NULL.
fn check_add_column_with_default_or_not_null(stmt: &Statement, diagnostics: &mut Vec<Diagnostic>) {
    let Statement::AlterTable { operations, name, .. } = stmt else { return };

    for op in operations {
        if let AlterTableOperation::AddColumn { column_def, .. } = op {
            let has_default = column_def.options.iter().any(|o| {
                matches!(o.option, ColumnOption::Default(_))
            });
            let has_not_null = column_def.options.iter().any(|o| {
                matches!(o.option, ColumnOption::NotNull)
            });

            if has_default || has_not_null {
                let issue = if has_default && has_not_null {
                    "DEFAULT and NOT NULL"
                } else if has_default {
                    "DEFAULT"
                } else {
                    "NOT NULL"
                };
                diagnostics.push(Diagnostic {
                    line: 1,
                    rule_id: "W002".to_string(),
                    message: format!(
                        "ALTER TABLE ADD COLUMN with inline {} (column '{}')",
                        issue, column_def.name
                    ),
                    suggestion: "Split into steps: 1) ADD COLUMN (type only), 2) UPDATE to set values, 3) ALTER COLUMN to add constraint.".to_string(),
                    is_error: false,
                });
            }
        }
    }
}

## Task 7: E002 — CREATE INDEX without ASYNC (span-based raw SQL check)

**Files:**
- Modify: `dsql-lint/src/lint.rs` (pass raw SQL to rules, use spans)
- Modify: `dsql-lint/src/rules/errors.rs`
- Modify: `dsql-lint/src/rules/mod.rs`

**Context:** `ASYNC` is a DSQL-specific keyword that `sqlparser` does not model in its `CreateIndex` AST node. A naive line-by-line raw SQL scan is fragile (misses multi-line statements, false-positives on comments). Instead, we use a **span-based hybrid approach**:

1. `sqlparser` parses the SQL and tells us *which* statements are `CreateIndex` and their **byte span** in the source
2. We extract the raw SQL slice for that span and check for `ASYNC` in that slice

This is robust against multi-line statements, comments, and string literals containing "CREATE INDEX".

**Step 1: Update `lint_sql` to use `parse_sql_with_spans`**

`sqlparser` supports `Parser::parse_statements` which returns `Vec<Statement>` each implementing the `Spanned` trait. The `Spanned` trait gives us `.span()` returning a `Span { start, end }` with byte offsets into the source.

use sqlparser::ast::Spanned; // trait for .span()

fn check_create_index(stmt: &Statement, raw_sql: &str, diagnostics: &mut Vec<Diagnostic>) {
    let Statement::CreateIndex(ref _create_index) = stmt else { return };

    // Use the Spanned trait to get the byte range of this statement in the source
    let span = stmt.span();
    let start = span.start.offset;
    let end = span.end.offset;

    // Extract the raw SQL slice for this statement
    let raw_slice = if end > start && end <= raw_sql.len() {
        &raw_sql[start..end]
    } else {
        // Fallback: can't determine span, check full SQL around this statement
        // This shouldn't happen with well-formed input
        return;
    };

    // Check if ASYNC appears in the raw SQL for this CREATE INDEX
    let upper = raw_slice.to_uppercase();
    let has_async = upper.contains("ASYNC");

    if !has_async {
        // Compute line number from byte offset
        let line = raw_sql[..start].chars().filter(|&c| c == '\n').count() + 1;

        diagnostics.push(Diagnostic {
            line,
            rule_id: "E002".to_string(),
            message: "CREATE INDEX must use ASYNC in DSQL".to_string(),
            suggestion: "Add ASYNC keyword: CREATE INDEX ASYNC ... or CREATE UNIQUE INDEX ASYNC ...".to_string(),
            is_error: true,
        });
    }
}
```

**IMPORTANT NOTES for implementing engineer:**

1. **Check `Spanned` availability.** The `Spanned` trait was added to sqlparser in recent versions. Run `cargo doc -p sqlparser --open` and search for `Spanned`. If the trait isn't available in v0.61, you have two fallback options:
   - **Option A:** Use `Parser::parse_sql_with_locations()` or similar API that preserves source locations
   - **Option B:** Split the raw SQL on `;` (respecting quotes) to get per-statement slices, then match each `Statement::CreateIndex` with its corresponding slice by index position. This is less precise but still much better than line-scanning.

2. **`ASYNC` as parser token.** `sqlparser` may reject `CREATE INDEX ASYNC ...` entirely since ASYNC is non-standard. Test this. If it fails to parse:
   - The statement won't appear in the AST at all
   - We may get a parse error (E000) instead
   - Consider pre-processing: strip `ASYNC` from raw SQL before parsing, record its positions, then check those positions in the rule. This is a known pattern for vendor-specific extensions.

Lint SQL files for [Amazon Aurora DSQL](https://aws.amazon.com/rds/aurora/dsql/) compatibility.

Parses SQL and reports errors (unsupported syntax) and warnings (best practice violations) with suggested fixes.

cargo install --path .

dsql-lint migration.sql [migration2.sql ...]
